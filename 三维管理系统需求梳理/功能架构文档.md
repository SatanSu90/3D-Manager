> **版本**: V3.0 | **日期**: 2026年7月 | **基于**: PRD V2.0 + 代码实现现状分析

---

## 一、文档概述

本文档基于已有 PRD V2.0，结合前后端代码实际实现情况，对平台功能架构进行全面梳理和完善。文档目标：

1. **明确功能边界**：每个模块做什么、不做什么（Non-goals）
2. **细化功能描述**：将 PRD 中的功能项拆解为可指导开发的颗粒度
3. **标注实现状态**：对照现有代码，标注每个功能模块的实现程度
4. **规划开发路径**：基于模块依赖关系和优先级，给出开发顺序建议

### 实现状态说明

| 标记 | 含义 | 说明 |
|------|------|------|
| ✅ 已实现 | 功能已完成，可投入使用 | 代码中已有完整实现 |
| ⚠️ 部分实现 | 核心框架已搭建，功能不完整 | 需要在此基础上扩展 |
| ❌ 未实现 | 尚未开始开发 | 需从零设计开发 |

## 二、功能架构总览

平台采用 **四层架构**，自底向上分别为基础支撑层、数据能力层、应用构建层、展示层。

### 2.1 架构层级

```
┌─────────────────────────────────────────────────────────┐
│  展示层    大屏展示 · 场景预览 · 报表查看 · 移动端适配      │
├─────────────────────────────────────────────────────────┤
│  应用构建层  场景编辑器(6大组件) · 场景管理 · 自定义报表    │
├─────────────────────────────────────────────────────────┤
│  数据能力层  数据接入 · 数据分析 · 三维模型库 · 资源库     │
├─────────────────────────────────────────────────────────┤
│  基础支撑层  用户管理 · 组织架构 · 角色权限 · 审计日志     │
└─────────────────────────────────────────────────────────┘
```

### 2.2 核心数据流

```
数据接入 → 数据分析 → 指标管理 → 场景编辑器(组件数据绑定) → 场景管理 → 大屏展示
                                    ↑                        ↑
                        三维模型库(模型拖入)          资源库(美化资源)
                                    ↑                        ↑
                              自定义报表(嵌入场景)    基础支撑层(权限/脱敏/跟踪)
```

## 三、各模块详细功能描述

### 3.1 基础功能模块

> **定位**: 平台级底层能力，为所有业务模块提供用户、权限、组织、安全等基础支撑。

#### 3.1.1 用户管理

| 功能项 | 描述 | 优先级 | 状态 | 开发要点 |
|--------|------|--------|------|----------|
| 用户注册 | 管理员创建用户账号（用户名/密码/邮箱/手机号） | P0 | ✅ | 后端已有 RegisterRequest，前端 LoginView 有注册入口 |
| 用户登录 | 账号密码登录，JWT Token 认证 | P0 | ✅ | 后端 AuthController + JwtTokenProvider 完整实现 |
| Token 刷新 | accessToken 过期自动刷新 | P0 | ✅ | 前端 request.ts 有并发刷新队列 |
| 用户信息修改 | 修改个人基本信息（头像/密码/联系方式） | P0 | ⚠️ | 后端有 UserController，但缺少个人信息修改接口 |
| 用户状态管理 | 启用/禁用用户账号 | P0 | ❌ | 需新增 User 实体 status 字段 + 管理接口 |
| 邮箱/手机号字段 | 用户实体增加 email、phone 字段 | P0 | ❌ | 当前 User 实体仅有 username/passwordHash/role/avatar |
| SSO 集成 | SAML/OAuth2/CAS 对接企业统一认证 | P2 | ❌ | 预留接口，后期集成 |
| 操作日志 | 记录用户关键操作行为 | P1 | ❌ | 需新增审计日志表 + AOP 切面 |

**开发建议**:
- 扩展 User 实体：增加 `email`、`phone`、`status`(ENABLED/DISABLED)、`lastLoginAt` 字段
- 新增 `PUT /api/users/me` 接口用于个人信息修改
- 新增 `PUT /api/users/{id}/status` 接口用于启用/禁用

#### 3.1.2 组织架构

| 功能项 | 描述 | 优先级 | 状态 | 开发要点 |
|--------|------|--------|------|----------|

| 部门排序 | 调整同级部门显示顺序 | P1 | ❌ | Department 实体增加 sortOrder 字段 |
| 部门导入 | Excel 批量导入部门与人员 | P2 | ❌ | 使用 Apache POI 解析 Excel |

**开发建议**:
- 新增 `Department` 实体：id, name, parentId, sortOrder, createdAt, updatedAt
- 新增 `user_department` 关联表（支持一人多部门）
- API: `/api/departments` (CRUD + tree)

#### 3.1.3 角色管理

| 功能项 | 描述 | 优先级 | 状态 | 开发要点 |
|--------|------|--------|------|----------|
| 角色定义 | 创建/编辑/删除角色 | P0 | ⚠️ | 当前仅有 ADMIN/USER 两种内置枚举角色，需改为角色表 |
| 权限配置 | 为角色配置功能权限/操作权限/数据权限 | P0 | ❌ | 新增 Permission 实体 + role_permission 关联表 |
| 角色分配 | 为用户分配一个或多个角色 | P0 | ❌ | 新增 user_role 关联表 |
| 内置角色 | 超级管理员/系统管理员/普通用户/访客 | P0 | ⚠️ | 当前仅 ADMIN/USER，需扩展为 4 种内置角色 |
| 权限继承 | 子角色继承父角色权限 | P2 | ❌ | 角色表增加 parentId |

**开发建议**:
- 将 `User.role` 枚举字段改为多对多关联 `User <-> Role`
- 新增 `Role` 实体：id, name, code, description, parentId, isBuiltin
- 新增 `Permission` 实体：id, name, code, type(MODULE/OPERATION/DATA), resourceId
- 新增 `role_permission` 关联表
- Spring Security 改为从数据库动态加载权限

#### 3.1.4 权限体系

| 权限维度 | 描述 | 状态 | 开发要点 |
|----------|------|------|----------|
| 模块权限 | 控制用户可访问的功能模块 | ❌ | 前端路由守卫 + 后端接口鉴权 |
| 操作权限 | 控制模块内的操作行为（查看/编辑/删除/控制） | ❌ | 后端 @PreAuthorize 注解 |
| 数据权限 | 控制可见数据范围（本部门/全量） | ❌ | MyBatis 拦截器或 JPA Specification |
| 场景预览权限 | 查看/操作模式分离 | ❌ | 场景实体增加 previewPermission 字段 |
| 所属权权限 | 私有/部门共享/全员共享 | ❌ | 所有资源实体增加 ownerId + visibility 字段 |

**Non-goals**: 不实现细粒度按钮级权限控制（成本过高），以模块+操作粒度为准。

#### 3.1.5 引用跟踪机制（全局）

| 功能项 | 描述 | 优先级 | 状态 | 开发要点 |
|--------|------|--------|------|----------|
| 引用关系记录 | 记录资源被引用的关系 | P0 | ❌ | 新增 reference_relation 表 |
| 删除保护 | 有引用时删除需二次确认 | P0 | ❌ | 后端删除接口增加引用检查逻辑 |
| 引用关系展示 | 列表展示引用详情 | P0 | ❌ | 前端增加引用详情弹窗 |
| 引用计数 | 列表中显示被引用次数 | P1 | ❌ | 后端 COUNT 查询 |
| 批量检查 | 批量操作前检查引用 | P1 | ❌ | 批量删除接口增加预检查 |
| 强删警告 | 强制删除时给出严重警告 | P1 | ❌ | 前端确认弹窗 + 后端 force 参数 |

**数据表设计**:
```sql
CREATE TABLE reference_relation (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    source_type VARCHAR(50) NOT NULL COMMENT '引用方类型: SCENE/REPORT/DASHBOARD',
    source_id BIGINT NOT NULL COMMENT '引用方ID',
    target_type VARCHAR(50) NOT NULL COMMENT '被引用类型: MODEL/METRIC/RESOURCE/INDICATOR',
    target_id BIGINT NOT NULL COMMENT '被引用ID',
    ref_position VARCHAR(200) COMMENT '引用位置描述',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_target (target_type, target_id),
    INDEX idx_source (source_type, source_id)
);
```

### 3.3 数据分析模块

> **定位**: 拖拽式分析工作台，将接入数据通过过滤、统计、聚合等操作转化为可复用指标，形成"分析→指标→报表/大屏"的数据价值链。

| 字段拖拽 | 拖入维度/指标区域，自动生成分析逻辑 | P0 | ❌ | 前端拖拽组件 (vuedraggable) |
| 过滤条件 | 等于/不等于/包含/范围/正则组合 | P0 | ❌ | 条件构建器 UI |
| 聚合运算 | SUM/COUNT/AVG/MAX/MIN/中位数/标准差 | P0 | ❌ | 后端动态 SQL 生成 |
| 分组统计 | 按维度分组，多级分组+小计/合计 | P0 | ❌ | GROUP BY 动态生成 |
| 排序与 Top N | 按指标排序，筛选 Top/Bottom N | P1 | ❌ | ORDER BY + LIMIT |
| 时间维度分析 | 同比/环比/滚动计算/时序预测 | P1 | ❌ | 时间函数计算 |

```sql
CREATE TABLE indicator (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(200) NOT NULL,
    description TEXT,
    analysis_task_id BIGINT NOT NULL COMMENT '关联分析任务',
    category_id BIGINT,
    owner_id BIGINT NOT NULL,

```

**交互要点**: 数据分析结果展示区提供"保存为指标"按钮 → 填写名称/分类/标签/共享范围 → 完成创建。指标修改时重新打开分析工作台编辑。

### 3.4 三维模型库模块

> **定位**: 管理数字孪生场景所需的三维模型资源，包括手工建模、倾斜摄影和点云数据。

#### 3.4.1 模型上传与管理

| 功能项 | 描述 | 优先级 | 状态 | 开发要点 |
|--------|------|--------|------|----------|
| 手工建模上传 | GLTF/GLB/OBJ/FBX/DAE 格式 | P0 | ⚠️ | 当前仅支持 GLB/GLTF，需扩展 Loader |

## 四、模块间关系与数据流

### 4.1 核心数据流链路

```
[数据接入管理] → 原始数据 → [数据分析] → 分析结果 → [指标管理] → 指标项
                                                                    ↓
                                                              绑定到组件
                                                                    ↓
[三维模型库] → 模型拖入 → [场景编辑器] ← 美化资源 ← [资源库]        ↓
                                ↓                                  ↓
                          保存场景                         绑定指标数据
                                ↓                                  ↓
                          [场景管理] ← 密码/权限 ← [基础功能]     ↓
                                ↓                                  ↓
                          [大屏展示] ← 嵌入 ← [自定义报表] ← 指标数据源
```

### 4.2 模块依赖关系

| 上游模块 | 下游模块 | 依赖关系 | 数据传递 |
|----------|----------|----------|----------|
| 基础功能 | 所有模块 | 权限/用户/组织/脱敏 | 用户ID、权限上下文 |
| 数据接入 | 数据分析 | 提供原始数据 | 数据源连接配置 |
| 数据接入 | 场景编辑器 | 直接数据绑定 | 数据源ID + 查询配置 |
| 数据分析 | 指标管理 | 分析结果沉淀 | analysisTask 关联 |
| 指标管理 | 场景编辑器 | 指标绑定组件 | indicatorId |
| 指标管理 | 自定义报表 | 指标作数据源 | indicatorId |
| 三维模型库 | 场景编辑器 | 模型拖入场景 | modelId |
| 资源库 | 场景编辑器 | 美化资源 | resourceId |
| 资源库 | 自定义报表 | 美化资源 | resourceId |
| 场景编辑器 | 场景管理 | 保存场景 | sceneData JSON |
| 场景管理 | 大屏展示 | 预览展示 | sceneId + 密码验证 |
| 自定义报表 | 场景编辑器 | 报表嵌入场景 | reportId |
| 自定义报表 | 报表查看 | 独立查看 | reportId |

### 4.3 关键交互链路

**链路1：数据到展示的完整闭环**
```
用户创建数据源 → 拖拽分析数据 → 保存为指标 → 在场景编辑器中绑定指标到图表组件
→ 配置刷新频率 → 保存场景 → 预览大屏 → 实时数据刷新
```

**链路2：模型到场景的完整闭环**
```
上传 3D 模型 → 自动压缩 → 在模型库预览 → 在场景编辑器中拖入模型
→ 调整位置/材质/动画 → 绑定数据驱动属性 → 保存场景 → 预览
```

**链路3：组态监控的完整闭环**
```
上传设备 3D 模型 → 在场景编辑器中放置设备 → 绘制管道连接
→ 绑定实时数据源 → 配置状态渲染规则 → 配置控制指令 → 预览操作
```

## 八、关键设计决策

### 8.1 场景数据存储方案

**决策**: 场景配置以 JSON 存储在 `scene.scene_data` 字段中

**理由**:
- 场景配置结构灵活，组件类型多样，关系型建模成本高
- JSON 支持嵌套结构，适合组件树表达
- MySQL 8.0 原生支持 JSON 类型，可做 JSON 查询

**约束**:
- scene_data 最大 16MB（MySQL JSON 类型限制）
- 组件 ID 使用 UUID 避免冲突
- 数据绑定配置单独抽取，便于引用跟踪

### 8.2 数据绑定架构

**决策**: 采用统一的数据绑定描述符

```json
{
  "sourceType": "DATABASE | API | FILE | INDICATOR | STATIC",
  "sourceId": 123,
  "queryConfig": {
    "sql": "SELECT * FROM ...",
    "params": {},
    "refreshInterval": 30
  },
  "fieldMapping": {
    "value": "$.data[0].temperature",
    "label": "$.data[0].name"
  },
  "masking": {
    "enabled": true,
    "rules": ["temperature"]
  }
}
```

**理由**: 统一描述符使所有组件共享同一套数据绑定引擎，降低开发和维护成本。

### 8.3 所属权与共享体系

**决策**: 所有资源实体统一增加 `owner_id` + `visibility` 字段

```sql
owner_id BIGINT NOT NULL,
visibility VARCHAR(20) DEFAULT 'PRIVATE' COMMENT 'PRIVATE/DEPARTMENT_SHARED/PUBLIC'
```

**查询规则**:
- PRIVATE: `WHERE owner_id = :currentUserId`
- DEPARTMENT_SHARED: `WHERE owner_id = :currentUserId OR (visibility = 'DEPARTMENT_SHARED' AND owner_id IN (部门成员))`
- PUBLIC: `WHERE owner_id = :currentUserId OR visibility != 'PRIVATE'`

### 8.4 组件系统架构

**决策**: 采用组件注册表 + 动态加载模式

```
ComponentRegistry
├── chart/         # 基础图表
│   ├── LineChart
│   ├── BarChart
│   └── ...
├── map/           # 地图
├── three-d/       # 三维
├── scada-2d/      # 2D组态
├── scada-3d/      # 3D组态
├── media/         # 媒体
└── report/        # 报表
```

每个组件实现统一接口:
```typescript
interface SceneComponent {
  type: string;              // 组件类型
  props: ComponentProps;     // 属性配置
  dataBinding?: DataBinding; // 数据绑定
  render(container: HTMLElement): void;  // 渲染
  destroy(): void;           // 销毁
}
```

## 九、Non-goals（明确不做什么）

1. **不做代码编辑器**: 平台定位为低代码，不提供在线代码编辑能力
2. **不做 AI 辅助生成**: V3.0 不包含 AI 自动生成场景/图表能力（后续版本规划）
3. **不做实时协同编辑**: 多人同时编辑同一场景不支持（成本过高）
4. **不做移动端编辑**: 移动端仅支持场景预览（只读），不支持编辑
5. **不做自定义组件开发框架**: 不提供用户自行开发组件的 SDK，仅支持导入 SVG/图片/脚本包
6. **不做细粒度按钮级权限**: 权限粒度到模块+操作级别，不到按钮级别
7. **不做数据治理**: 数据治理由企业数据中台承担，平台仅做轻量数据接入
8. **不做物理引擎**: 三维场景不集成物理引擎（碰撞检测/重力模拟），组态动画通过预设路径实现
