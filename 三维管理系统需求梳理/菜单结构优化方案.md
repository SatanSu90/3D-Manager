# 3D-Manager 三维数字孪生平台 — 侧边栏菜单结构优化方案

> **版本**: V1.0 | **日期**: 2026年7月 | **编制**: 产品经理  
> **背景**: 用户反馈当前侧边栏菜单"分类不符合信息化系统的菜单分类，感觉是拼凑的内容"

---

## 一、现状问题分析

### 1.1 当前菜单结构回顾

```
AppSidebar.vue 当前结构：
├── 顶部导航（平铺，无分组）
│   ├── 模型库 → /
│   ├── 场景管理 → /scenes
│   ├── 数据源管理 → /datasources
│   └── 部门管理 → /departments (admin only)
├── 分类筛选（模型库页面级功能，嵌入全局侧边栏）
│   ├── 全部模型
│   └── 分类树（动态加载）
├── 标签筛选（模型库页面级功能，嵌入全局侧边栏）
│   └── 标签列表（动态加载）
└── 数据管理（底部小字区域）
    ├── 分类管理 → /categories
    └── 标签管理 → /tags
```

### 1.2 逐条问题清单

| 序号 | 问题 | 具体表现 | 影响 |
|------|------|----------|------|
| 1 | **页面级筛选混入全局导航** | 分类筛选、标签筛选是模型库页面的局部功能，却占据了全局侧边栏约 50% 的空间 | 侧边栏职责不清，用户在其他页面（场景管理、数据源管理）看到无用的模型筛选器，产生困惑 |
| 2 | **缺少分组层次** | 7 个菜单项全部平铺，没有功能域分组，无法体现模块化架构 | 不符合信息化系统"按功能域分组"的通用范式，用户无法快速定位功能 |
| 3 | **命名不统一** | "模型库"（无"管理"后缀）vs "场景管理""数据源管理"（有后缀）；"数据管理"分组名与实际内容（分类/标签）不匹配 | 视觉和语义不一致，显得随意拼凑 |
| 4 | **场景编辑器入口缺失** | `/editor` 路由存在但侧边栏无入口，用户只能从场景管理列表间接进入 | 核心功能入口隐蔽，新用户难以发现 |
| 5 | **管理后台入口缺失** | `/admin` 路由存在但侧边栏无入口，管理员需手动输入 URL | 管理功能不可达 |
| 6 | **权限分层不清晰** | "部门管理"是 admin only，但与普通功能混在同一区域，无视觉隔离 | 权限边界模糊，普通用户看到无权限菜单项产生疑惑 |
| 7 | **无未来扩展预留** | 当前仅 6 个菜单项，系统规划了 8 大模块 35 子模块，新增功能无处安放 | 后续每加一个功能都需要重新调整菜单结构 |
| 8 | **分组名称语义偏差** | 底部"数据管理"实际只包含分类管理和标签管理，与"数据"无直接关系 | 误导用户，分类/标签是模型资源的元数据管理，不是数据管理 |

## 二、设计原则

信息化系统菜单设计应遵循以下原则：

| 原则 | 说明 | 本方案应用 |
|------|------|------------|
| **P1 功能聚合** | 相关功能归入同一分组，按业务域而非技术实现划分 | 按四层架构划分：资源管理、场景应用、数据中心、资源报表、系统管理 |
| **P2 层次清晰** | 采用"分组 → 菜单项"二级结构，不超过三级 | 侧边栏仅二级：一级为功能分组，二级为具体页面 |
| **P3 命名统一** | 所有菜单项遵循统一命名规范，避免混用 | 管理类页面统一"XX管理"，工作台类用"XX编辑/XX分析" |
| **P4 权限隔离** | 管理类功能与业务类功能视觉分离，按权限控制可见性 | "系统管理"分组置于底部，仅 admin 可见；其余分组所有用户可见 |
| **P5 局部全局分离** | 页面级筛选、操作不放全局导航，由页面自行承载 | 模型分类/标签筛选移至模型管理页面内部 |
| **P6 渐进扩展** | 预留扩展分组和菜单位，新模块按规则接入即可 | 每个分组预留未来菜单项（标记为"待开发"），新增模块只需添加分组 |
| **P7 图标一致** | 同一分组使用风格统一的图标，全平台图标来源统一 | 统一使用 lucide-vue-next 图标库，分组用 16px，菜单项用 14px |

### 4.1 不纳入侧边栏的路由

以下路由为详情页/全屏页/子页面，通过页面内跳转进入，不直接出现在侧边栏：

| 路由 | 页面 | 进入方式 |
|------|------|----------|
| `/model/:id` | 模型详情 | 从"模型管理"列表点击进入 |
| `/editor/:sceneId` | 加载特定场景编辑器 | 从"场景管理"列表点击"编辑"进入 |
| `/preview/:sceneId` | 场景预览（全屏） | 从"场景管理"列表点击"预览"新窗口打开 |
| `/login` | 登录页 | 未登录时自动跳转 |

### 4.2 模型库页面本地筛选方案

将当前侧边栏中的"分类筛选"和"标签筛选"迁移至模型管理页面内部，作为页面级筛选面板：

```
模型管理页面 (/models)
├── 页面顶部：搜索栏 + 操作按钮（上传模型等）
├── 页面左侧筛选面板（可折叠）
│   ├── 分类筛选（分类树，支持多级展开）
│   └── 标签筛选（标签云，支持多选）
└── 页面主区域：模型卡片/列表网格
```

**实现要点**：
- 筛选状态通过 URL Query 参数持久化（如 `/models?categoryId=1&tagIds=2,3`），支持分享和书签
- 筛选面板可折叠/展开，默认展开
- 筛选逻辑保持不变，仅迁移组件位置

## 五、命名规范

### 5.1 菜单项命名规则

| 类型 | 命名模式 | 示例 | 说明 |
|------|----------|------|------|
| 列表/管理类页面 | `XX管理` | 模型管理、场景管理、数据源管理、用户管理 | CRUD 管理页面统一加"管理"后缀 |
| 编辑器/工作台类页面 | `XX编辑` / `XX分析` | 场景编辑、数据分析 | 工作型页面用动词后缀 |
| 配置类页面 | `XX配置` / `XX权限` | 系统配置、角色权限 | 配置型页面用对应后缀 |
| 日志/监控类页面 | `XX日志` | 操作日志 | 日志型页面用"日志"后缀 |

### 5.2 分组命名规则

| 规则 | 说明 | 示例 |
|------|------|------|
| 使用名词短语 | 不用动词，不用句子 | "资源管理"而非"管理资源" |
| 2-4 个汉字 | 简洁明了 | "数据中心""场景应用""系统管理" |
| 避免与菜单项同名 | 分组名是域，菜单项是具体功能 | 分组"资源管理"，菜单项"模型管理"（不重复） |

### 5.3 当前命名修正对照

| 原名称 | 修正后 | 原因 |
|--------|--------|------|
| 模型库 | 模型管理 | 统一"管理"后缀，"库"字偏存储含义 |
| 数据管理（分组名） | 资源管理 | 原名与内容（分类/标签）语义不符，分类标签属于模型资源元数据 |
| —（无分组） | 场景应用、数据中心、资源报表、系统管理 | 新增分组，实现功能聚合 |

### 6.4 待开发菜单项的显示策略

待开发菜单项有两种显示策略，建议采用**策略 B**：

| 策略 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| A. 隐藏 | 待开发菜单项不显示在侧边栏 | 界面干净 | 用户不知道未来有什么功能 |
| **B. 置灰显示** | 显示但灰色不可点击，鼠标悬停提示"即将上线" | **用户可预知功能全貌，体现产品规划** | 需额外处理禁用状态 |

**实现方式**：菜单配置增加 `status: 'active' | 'coming'` 字段，`coming` 状态的菜单项渲染为灰色禁用，悬停显示 Tooltip "即将上线"。

## 七、扩展性说明

### 7.1 新增菜单项

当未来开发新功能时，按以下步骤接入菜单：

1. 确定功能所属分组（资源管理/场景应用/数据中心/资源报表/系统管理）
2. 在该分组下添加菜单项配置
3. 在 `router/index.ts` 注册路由
4. 开发对应 View 组件
5. 将菜单项 `status` 从 `coming` 改为 `active`

**示例**：M2 阶段开发"数据分析"功能
```
步骤1: 确认属于"数据中心"分组
步骤2: 菜单项已预埋，将 status 从 'coming' 改为 'active'
步骤3: 确保 /analysis 路由已注册
步骤4: 开发 AnalysisView.vue
```

### 7.2 新增功能分组

若未来出现不属于现有 5 个分组的新模块，按以下规则新增分组：

1. 在侧边栏现有分组之后、系统管理之前插入新分组
2. 分组名遵循命名规范（2-4 字名词短语）
3. 选择风格统一的 lucide 图标
4. 配置分组权限（默认所有用户，管理类设为 admin）

**预留分组位**：根据功能架构文档，当前 5 个分组已覆盖 8 大模块。若后续出现"增强优化"模块（性能监控、缓存管理、API 网关、消息通知），可新增"运维监控"分组置于"系统管理"之前。
